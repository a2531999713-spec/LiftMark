# 练刻 LiftMark 小组训练完整方案

## 1. 小组训练定位

小组训练是练刻的核心能力，不是附属功能。

产品定位：

```
让 2-4 人可以一起执行同一个训练计划，每个人拥有独立参数、独立重量、独立记录、独立分析，同时保留小组协作、同练、统计、激励和共享能力。
```

小组训练要同时支持三种真实场景：

|场景|用户行为|产品能力|
|---|---|---|
|本机小组训练|一台手机记录多人数据|适合线下一起练|
|在线多人同练|每个人用自己的手机同步训练|适合多设备、远程同练、教练带练|
|训练后上传小组|先本地记录，训练后选择同步|适合网络差、临时搭子、隐私控制|

当前产品文档里已经定义了“本地小组”：适合同一台设备多人轮换记录，数据保存在本机；后续再支持账号登录、邀请成员、多设备同步和授权共享数据。 这套方案是在这个基础上，把最终完整形态一次设计清楚。

---

## 2. 小组的核心概念

### 2.1 小组

小组是训练协作单位。

一个小组包含：

|内容|说明|
|---|---|
|小组名称|例如“周一四练小队”“宿舍增肌组”“深蹲三人组”|
|小组成员|2-4 人为主要体验目标，可根据会员权益扩展|
|当前计划|小组正在执行的训练计划|
|当前周期|增力、增肌、减量、自由训练等|
|当前周数|当前处于计划第几周|
|小组训练记录|小组共同完成的训练 session|
|小组统计|出勤、训练量、完成率、PR、连续训练|
|小组权限|谁能管理成员、计划、记录、邀请、数据可见范围|

小组不是简单的“好友列表”，而是一个围绕训练计划执行的协作空间。

---

### 2.2 成员

成员是小组内的训练个体。

每个成员必须有独立数据：

|数据|说明|
|---|---|
|昵称|小组内显示名称|
|头像|可选|
|体重|可选，用于部分分析|
|1RM|卧推、深蹲、硬拉、推举等|
|加重单位|杠铃、哑铃、器械的最小加重单位|
|训练历史|每个人独立保存|
|进阶建议|每个人独立生成|
|恢复状态|每次训练前可独立选择|
|数据共享权限|决定小组能看到多少数据|

产品文档也明确把 member 模块作为多人建议重量和训练记录归属的基础，小组模块绑定当前计划、周期、周数等训练状态。

---

### 2.3 本机成员与账号成员

小组成员分两类：

|成员类型|说明|
|---|---|
|本机成员|没有登录账号，只存在当前设备中|
|账号成员|登录后有云端身份，可以跨设备同步|

本机成员适合线下训练搭子快速使用。账号成员适合长期同步、云备份、在线同练、数据分析和会员权益。

关键规则：

```
本机成员的数据默认只保存在当前设备。账号成员的数据可以同步到云端。本机成员后续可以绑定账号并认领历史记录。
```

---

## 3. 小组人数规则

建议默认小组人数仍围绕 2-4 人设计，因为力量训练现场多人轮换超过 4 人后效率会明显下降，产品目标也是支持 2-4 人一起查看训练、计算重量和记录结果。

建议规则：

|用户类型|小组人数|说明|
|---|---|---|
|免费用户|1-2 人|可体验搭子训练|
|Pro 用户|最多 4 人|适合真实训练搭子|
|永久会员|最多 4 人|与 Pro 一致，避免滥用|
|教练/团队版|可超过 4 人|用于私教、小团队、训练营|

普通用户和 Pro 用户不要轻易放开到 6-10 人，因为这会把产品从“训练现场工具”变成“团队管理工具”，复杂度会上升很多。4 人是比较合理的默认上限。

---

## 4. 小组角色与权限

### 4.1 角色设计

|角色|权限|
|---|---|
|组长 owner|创建小组、管理成员、设置计划、管理小组权限、处理成员申请|
|成员 member|参与训练、记录自己的数据、查看被授权的小组数据|
|教练 coach|分配计划、查看成员训练、给出建议，不一定参与训练|
|访客 guest|只读查看部分内容，不能记录或修改|

产品文档里已经保留了 owner、member、coach、guest 这些角色模型。

### 4.2 权限矩阵

|功能|组长|成员|教练|访客|
|---|---|---|---|---|
|修改小组名称|可以|不可以|可选|不可以|
|邀请成员|可以|可选|可选|不可以|
|移除成员|可以|不可以|可选|不可以|
|设置当前计划|可以|可选|可以|不可以|
|开始小组训练|可以|可以|可以|不可以|
|记录自己的训练|可以|可以|可选|不可以|
|记录他人训练|可授权|可授权|可授权|不可以|
|查看个人完整数据|本人授权后|仅本人|授权后|不可以|
|查看小组汇总|可以|可以|可以|可选|
|删除训练记录|记录创建者/组长|仅自己的|可授权|不可以|

## 5. 小组训练模式

小组训练不要只做一个“开始训练”按钮，而应该在开始前选择训练方式。

### 5.1 本机小组训练

本机小组训练是一个手机记录多人数据。

适合：

|场景|说明|
|---|---|
|两三个人在同一个健身房|一个人拿手机，大家轮流报重量和次数|
|临时搭子训练|不需要每个人都登录|
|网络差的健身房|不影响训练记录|
|用户不想注册|也可以完整使用核心训练功能|

核心规则：

```
一个设备创建一场训练 session。同一场训练中包含多个成员。每个成员有独立重量、次数、RPE/RIR、完成状态。数据先写入本地 SQLite。训练后可以选择是否同步到云端小组。
```

训练中必须保持低延迟。本项目技术架构本来就是本地优先，训练数据以 Expo SQLite 写入，UI 只负责展示和输入。 训练执行页也要求每次修改立即保存 SQLite，避免真实训练现场丢数据。

---

### 5.2 在线多人同练

在线多人同练是每个成员用自己的手机进入同一场训练。

适合：

|场景|说明|
|---|---|
|每个人都有账号|数据归属清晰|
|不在同一个地点|可以远程同练|
|教练带小组|教练可以实时看完成情况|
|小组挑战|可以统计完成率、出勤和进度|
|长期训练搭子|每个人都有自己的历史分析|

核心规则：

```
组长或成员创建在线训练房间。小组成员加入训练房间。每个人记录自己的组。当前动作、完成进度、训练状态实时同步。训练结束后生成个人总结和小组总结。
```

在线多人同练需要云服务支持。现有架构已预留 Supabase Auth、PostgreSQL、Storage、Edge Functions 和 Realtime 作为后续候选。

---

### 5.3 训练后上传小组

训练后上传是独立于训练模式的数据同步能力。

适合：

|场景|说明|
|---|---|
|先离线训练|训练时不受网络影响|
|一个手机记录多人|训练后把数据同步给对应成员|
|成员未登录|可先保存为本机成员，后续再认领|
|用户重视隐私|可以选择只上传自己的数据|

上传选项：

|选项|说明|
|---|---|
|不上传|只保存在本机|
|只上传我的数据|只同步当前登录用户的记录|
|上传小组汇总|只上传完成率、训练量等汇总|
|上传完整小组记录|上传动作、重量、次数、RPE/RIR|
|等成员确认后上传|涉及他人完整数据时需要授权|

这个功能很重要，因为它既保留了本地训练的流畅性，又给小组分析和云备份留出空间。

---

## 6. 开始训练流程

点击“开始训练”后，进入训练方式选择页。

```
开始训练-> 选择训练方式   -> 本机小组训练   -> 在线多人同练   -> 单人训练-> 选择参与成员-> 确认今日计划-> 选择恢复状态-> 进入训练执行页
```

### 6.1 选择训练方式

页面文案建议：

```
选择训练方式本机小组训练用这一台手机记录多人数据，离线也能练。在线多人同练邀请小组成员一起进入训练，训练进度实时同步。单人训练只记录自己的训练。
```

### 6.2 选择参与成员

支持以下操作：

|操作|说明|
|---|---|
|勾选本次参与成员|不一定每次小组成员都全部训练|
|添加临时成员|适合临时搭子|
|切换当前记录人|本机小组训练时常用|
|查看成员建议重量|每个人独立计算|
|标记缺席|小组统计可用|

### 6.3 确认今日计划

今日训练要读取小组当前用户计划、周期、周数，再按恢复状态过滤动作，并按成员计算建议重量。现有产品业务流程就是这样定义的。

确认页显示：

|内容|说明|
|---|---|
|当前计划|例如“四练增力增肌”|
|当前周期|增力 / 增肌 / 减量|
|当前周数|第几周|
|今日训练日|Day 1 / Day 2 / Day 3 / Day 4 / 补弱 / 自由训练|
|今日动作|A/B/C 优先级|
|参与成员|本次训练成员|
|每人建议重量|按每个人 1RM 和加重单位计算|

## 7. 训练执行页设计

训练执行页是小组训练的核心，不能做成 Excel 表格。产品文档也明确要求训练执行页从训练页进入，不显示底部 Tab，使用沉浸式页面，突出当前动作、多成员记录卡、重量/次数、RPE/RIR、完成本组和自动保存状态。

### 7.1 页面结构

训练执行页建议结构：

```
顶部：当前动作 / 当前进度 / 退出或暂停中部：动作说明本动作计划组数、次数、强度、休息时间成员记录卡底部：完成本组跳过动作替换动作休息倒计时
```

### 7.2 成员记录卡

每个成员一张卡片：

|字段|说明|
|---|---|
|成员昵称|当前记录对象|
|建议重量|根据个人 1RM 计算|
|实际重量|可手动调整|
|实际次数|可手动调整|
|RPE|6-10 点选，可为空|
|RIR|0-5 点选，可为空|
|完成状态|未开始 / 已完成 / 跳过|
|历史参考|上次同动作表现|
|本组备注|可选|

RPE/RIR 不建议用键盘输入，应该点选。产品文档已明确 RPE/RIR 改为 preset 点选，RPE 为 6/7/8/9/10/清空，RIR 为 0-5/清空。

---

## 8. 小组训练记录方式

### 8.1 按动作轮换

默认方式：

```
动作 1：卧推  张三 第 1 组  李四 第 1 组  王五 第 1 组  张三 第 2 组  李四 第 2 组  王五 第 2 组动作 2：划船  张三 第 1 组  李四 第 1 组  王五 第 1 组
```

适合力量训练现场，因为多人通常共用器械轮换。

### 8.2 按成员记录

可选方式：

```
张三完成卧推全部组李四完成卧推全部组王五完成卧推全部组
```

适合器械不冲突、每个人节奏不同的训练。

### 8.3 自由记录

自由记录用于临时训练：

|能力|说明|
|---|---|
|临时添加动作|不在计划里也能记录|
|临时加组|状态好时多做|
|临时删组|时间不够或疲劳|
|跳过动作|器械占用或身体不适|
|替换动作|保留原计划动作快照|

产品文档中也要求器械被占时可替换动作，但原计划动作不变；历史记录要保存当时的计划快照。

---

## 9. 动作替换规则

动作替换是小组训练的高频功能。

### 9.1 替换场景

|场景|示例|
|---|---|
|器械被占|杠铃卧推改哑铃卧推|
|身体不适|深蹲改腿举|
|人太多|固定器械改自由重量|
|训练目标调整|高疲劳时换轻量动作|
|健身房设备差异|没有哈克深蹲机，改腿举|

### 9.2 替换规则

```
替换只影响本次训练。原计划不被修改。训练记录保存替换后的动作。历史记录保留原计划动作字段。
```

对应数据关系：

|字段|说明|
|---|---|
|planned_exercise_id|原计划动作|
|exercise_id|实际执行动作|
|replaced_from_exercise_id|从哪个动作替换而来|
|replacement_reason|替换原因，可选|
|replacement_note|备注，可选|

---

## 10. 训练中断、退出和恢复

小组训练必须支持中途退出。

### 10.1 中断场景

|场景|产品处理|
|---|---|
|用户切到微信|训练继续保留|
|锁屏|回来后恢复训练页|
|App 被系统杀掉|从本地 session 恢复|
|网络断开|本机继续记录，稍后同步|
|用户误点退出|必须二次确认|
|训练未完成|保存为进行中|

技术架构里已经要求 SQLite 写入失败必须提示，训练中断后应能从 SQLite 恢复，Repository 不应吞掉错误。

### 10.2 退出提示

用户点击退出训练时提示：

```
是否结束本次训练？继续训练保存并退出放弃本次训练
```

“放弃本次训练”必须二次确认。

---

## 11. 数据同步规则

### 11.1 同步原则

小组训练同步必须遵守四个原则：

```
本地先保存用户可控制他人数据需授权云端只做同步和分析，不影响训练现场记录
```

当前技术架构已预留 syncQueue、syncTypes、conflictResolver，并建议后续同步字段包括 remote_id、sync_status、deleted_at。

### 11.2 同步状态

每条训练记录应有同步状态：

|状态|说明|
|---|---|
|local_only|仅本地|
|pending_sync|等待同步|
|synced|已同步|
|sync_failed|同步失败|
|conflict|有冲突，需要处理|
|deleted_pending|本地删除，等待云端同步|

### 11.3 同步对象

|数据|是否同步|说明|
|---|---|---|
|小组信息|同步|名称、成员、权限|
|成员基础信息|同步|昵称、头像、角色|
|成员 1RM|授权后同步|属于个人敏感训练数据|
|训练 session|同步|小组训练记录|
|每组训练数据|授权后同步|重量、次数、RPE/RIR|
|小组汇总|同步|可用于排名和成就|
|本机游客成员|默认不同步|绑定账号后可认领|
|计划文件|可同步|计划不应包含成员 1RM 或训练记录|

计划导入导出文件本身不应包含训练记录、成员 1RM 或个人参数，这一点产品文档已经明确。

---

## 12. 数据权限与隐私

训练数据包含个人能力、体重、训练表现等信息，不能默认全部公开。技术架构也明确提醒本地训练数据包含体重、训练表现等隐私信息，后续云同步需要处理账号、权限、删除和冲突。

### 12.1 数据可见等级

|等级|小组能看到什么|
|---|---|
|仅自己可见|小组看不到个人训练详情|
|只共享完成状态|看到是否完成训练|
|共享汇总数据|看到训练次数、训练量、完成率|
|共享完整记录|看到动作、重量、次数、RPE/RIR|
|教练可见|教练可以查看完整记录并给建议|

### 12.2 默认建议

|用户类型|默认权限|
|---|---|
|普通小组成员|共享汇总数据|
|本机成员|仅本机可见|
|教练小组成员|教练可见，但加入时必须确认|
|临时成员|仅本次训练可见|
|未成年人或敏感用户|保守显示，不做公开排名|

### 12.3 删除和退出

|操作|规则|
|---|---|
|成员退出小组|保留自己的个人数据|
|组长移除成员|小组不再显示该成员后续数据|
|删除小组训练|需要组长或记录创建者权限|
|删除个人记录|用户可以删除自己的记录|
|清空本地数据|必须二次确认|

---

## 13. 小组分析

小组分析要和个人分析分开，不能混用。产品文档已经强调记录页默认统计当前成员个人数据，小组汇总未实现前不能把个人数据和小组数据混在一起。

### 13.1 个人分析

个人分析包括：

|模块|内容|
|---|---|
|本周训练次数|本人完成几次训练|
|本周训练量|本人训练总量|
|完成组数|本人完成多少组|
|动作趋势|卧推、深蹲、硬拉等|
|预估 1RM|根据历史表现推算|
|PR|重量 PR、次数 PR、训练量 PR|
|疲劳提示|根据最近训练表现给出提醒|
|下次建议|加重、维持、降重、减量、加次数|

### 13.2 小组分析

小组分析包括：

|模块|内容|
|---|---|
|小组本周训练次数|小组完成几场训练|
|小组出勤率|成员参与情况|
|小组完成率|计划动作完成比例|
|小组训练量|小组总训练量|
|同练次数|几个人一起完成训练|
|小组 PR|本周谁刷新了 PR|
|小组连续训练|小组连续多少周训练|
|落后提醒|哪些成员训练缺席较多|
|计划进度|当前计划完成到哪里|

小组分析不要默认做“谁最强排行榜”，因为这样容易诱导冲重量。更适合做：

```
谁最稳定谁连续训练最多谁完成率最高谁刷新了自己的 PR小组是否按计划完成
```

## 14. 小组奖励与成就

奖励机制应围绕坚持、完成、进步和恢复，不应该只奖励重量最大。

### 14.1 个人成就

|成就|触发条件|
|---|---|
|第一次刻下训练|完成首次训练|
|连续两周训练|连续 2 周有训练记录|
|完成一个周期|完成完整训练周期|
|卧推新 PR|卧推动作刷新个人记录|
|深蹲新 PR|深蹲动作刷新个人记录|
|训练记录完整|连续多次填写 RPE/RIR|
|恢复意识|按建议完成减量或休息|
|稳定执行者|计划完成率达到一定比例|

### 14.2 小组成就

|成就|触发条件|
|---|---|
|小组首次同练|2 人以上完成同一场训练|
|全员出勤|小组成员全部完成本周计划|
|连续同练|连续多周一起训练|
|小组完成一个周期|小组完成完整训练周期|
|全员 PR 周|多名成员刷新个人 PR|
|不鸽训练|小组连续多周保持训练|
|最佳搭子|两人多次一起完成训练|

### 14.3 挑战机制

|挑战|说明|
|---|---|
|本周完成挑战|本周每人完成指定训练次数|
|小组总训练量挑战|小组累计训练量达到目标|
|动作专项挑战|卧推、深蹲、硬拉专项|
|连续训练挑战|连续 N 周保持训练|
|恢复挑战|按计划完成减量周|

挑战要避免直接鼓励“谁重量最大”，可以用“个人进步率”“完成率”“连续性”替代绝对排名。

## 17. 小组计划规则

### 17.1 小组当前计划

每个小组可以绑定一个当前计划。

规则：

```
小组当前计划用于生成今日训练。成员仍保留个人参数。同一个计划下，每个人建议重量不同。计划修改不能破坏历史记录。
```

产品文档已经明确：系统方案不是用户计划，用户必须点击“使用此方案”后才生成自己的训练计划，训练记录不能直接绑定系统方案，只能绑定用户计划或训练时计划快照。

### 17.2 计划适配小组

小组计划需要支持：

|能力|说明|
|---|---|
|同计划不同重量|按每个人 1RM 计算|
|成员跳过动作|个人可跳过，不影响他人|
|成员替换动作|替换只影响个人或本次训练|
|小组统一进度|默认所有成员跟随同一周期|
|个人偏移|可允许某成员补练或落后|
|计划快照|训练记录保存当时计划内容|

## 18. 成员状态与恢复

小组训练前，每个成员可以选择恢复状态。

状态：

|状态|训练调整|
|---|---|
|状态好|完整显示 A/B/C 动作|
|一般|删除或弱化 C 动作|
|状态差|只保留 A 动作或建议减量|
|很差|建议休息或恢复训练|
|自定义|用户手动调整动作和重量|

产品文档里已经定义了恢复建议状态，包括 normal、remove_c、only_a、reduce_weight、deload、rest。

小组训练中要注意：

```
小组可以一起练同一个动作，但每个人可以有不同训练量。不要因为小组进度强迫状态差的人完成全部训练。
```

## 23. 防滥用和数据真实性

小组奖励和分析会遇到刷数据问题，不需要一开始强控制，但要有规则。

### 23.1 风险

|风险|处理|
|---|---|
|用户乱填重量|不做强校验，但分析中标记异常|
|刷训练量|成就降低绝对训练量权重|
|代替别人上传数据|他人完整数据需确认|
|小组拼单滥用|限制小组数量和成员上限|
|删除失败记录|删除保留操作记录或软删除|
|排行榜刺激冲重量|不做绝对重量排行榜|

### 23.2 异常提示

例如：

```
本次重量相比最近记录提升较大，请确认是否记录正确。
```

不要用惩罚语气，只做训练安全和数据准确提示。

## 27. 关键产品原则

最后把小组训练的原则定死：

```
1. 小组训练不是简单多人列表，而是多人一起执行训练计划。2. 每个人的数据必须独立：独立 1RM、独立重量、独立记录、独立建议。3. 本机小组训练必须保留，一台手机记录多人是核心场景。4. 在线多人同练是增强能力，不应该破坏本地训练流畅性。5. 训练现场永远本地优先，不能因为网络影响记录。6. 个人分析和小组分析必须分开，不能混用数据。7. 上传他人完整训练数据必须授权。8. 小组奖励鼓励坚持、完成和个人进步，不鼓励盲目冲重量。9. 计划是数据，不是代码；训练记录必须保存计划快照。10. 训练记录不能用 AsyncStorage，必须走 SQLite / Repository / Sync 边界。
```

这些原则也和现有项目规则一致：计划是数据、计划模板和个人参数分离、计划和训练记录分离、多人数逻辑保留、本地 SQLite 优先、Domain 层不依赖 UI、训练记录不能用 AsyncStorage。

## 结论

最终的小组训练方案应该是：

```
本机小组训练 + 在线多人同练 + 训练后同步 + 小组分析 + 成就激励 + 数据权限
```
