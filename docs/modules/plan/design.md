# Plan 模块设计文档

更新时间：2026-06-12

## 1. 设计目标

计划模块要支撑“系统方案库”和“我的计划”分离：系统方案用于参考和复制，用户计划才是真正执行、编辑、导出和记录训练的对象。

## 2. 核心对象

| 对象 | 定位 | 是否可执行 | 是否可编辑 | 是否绑定训练记录 |
|---|---|---|---|---|
| SystemTrainingScheme | 系统只读方案目录 | 否 | 否 | 否 |
| UserTrainingPlan | 用户本地训练计划 | 是 | 是 | 是 |
| ActivePlan | 当前小组正在执行的计划状态 | 是 | 通过 UserTrainingPlan 修改 | 训练开始时生成快照 |
| TrainingRecord | 实际训练记录 | 不适用 | 不适用 | 绑定 UserTrainingPlan 或快照 |

## 3. 页面结构

计划页结构固定为：

```text
计划
- 当前计划
- 我的计划
- 本周安排
- 系统方案
- 创建计划
- 导入计划
```

系统方案卡片显示目标、适合人群、频率、周期、训练日结构、简介、标签和可用状态。可用方案提供“查看详情”和“使用此方案”。开发中的方案显示“开发中”，不能无响应。

## 4. 核心业务规则

- 系统方案不是用户计划。
- 系统方案不会直接出现在“我的计划”中。
- 用户点击“使用此方案”后，才复制生成新的用户计划。
- 复制出的用户计划 `source` 为 `system_copy`，并记录 `originSchemeId`。
- 当前计划只能指向用户计划；训练页读取的是 `groups.active_plan_id` 指向的用户计划。
- 训练记录不能直接绑定系统方案。
- 导出对象是用户计划，不导出系统方案、训练记录、成员 1RM、身体数据、激活码或设备信息。
- 导入计划必须重新生成本地 ID，不覆盖已有计划。

## 6. 扩展点

第一阶段只做本地系统方案库。后续可以接远程目录：

```text
remote scheme catalog -> catalog.json -> scheme package url -> 下载 -> 使用此方案 -> 复制为 UserTrainingPlan
```

后台可扩展管理员上传、设置方案标题/目标/难度/标签、发布 catalog 和 App 拉取方案列表。
