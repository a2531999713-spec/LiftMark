## 2026-06-24 手机号密码登录与验证码绑定重构

- 首次使用必须完成账号登录 / 注册，未登录不能进入首页、计划、记录、我的等主 Tab。
- 登录主路径为手机号 + 密码；不再把手机号验证码作为唯一登录方式。
- 注册必须使用手机号 + 验证码 + 密码 + 昵称，验证码用于验证并绑定手机号。
- 验证码后续也用于手机号绑定 / 换绑、找回密码等敏感流程，不作为“无密码自动注册”的主路径。
- 启动状态分层：`checking`、`unauthenticated`、`authenticated`、`offline_authenticated`。
- 有本地 session 时，即使服务器不可达，也允许进入默认主界面并显示“当前离线，已进入本机模式”。
- 离线进入时不主动拉取云端完整资料，不做全量同步，不清空 session，不清空 SQLite。
- 登录免费版可以真实训练、离线训练并保存本地 SQLite 记录；免费版限制集中在 `src/domain/auth/access-control.ts`，不散落到页面组件。
- Pro / 永久会员解锁高级历史、完整小组、高级分析、在线同练和完整云同步等权益入口；未接支付的购买动作必须显示开发中提示。
- 登录页是双态认证表单：密码登录展示手机号、密码和协议勾选；注册展示手机号、验证码、密码、昵称和协议勾选。
- 当前 UI 不展示继续浏览、预览模式、验证码一键登录、第三方登录、会员权益或小组协作卖点。
- 验证码发送失败必须转换为中文提示，不能向用户展示 `fetch failed`、`ConnectException`、服务器 IP 或 stack trace。
- 退出登录只清除账号 session，不清空本机 SQLite 训练数据；训练执行仍然本地优先，服务器失败不影响现场记录。

## 1. 产品定位

“练刻 / LiftMark”是一个支持多人同练、计划自定义、自动重量计算、训练记录、周期进阶和计划分享的力量训练 App。

第一版不是普通单人训练记录工具，也不是 Excel 表格搬运，而是安卓本地可用的多人训练执行 App。

## 2. 用户角色

| 角色 | 需求 |
|---|---|
| 普通健身者 | 按计划训练、记录数据、获得下次建议 |
| 健身搭子 | 2-5 人一起练，同动作不同重量，轮流记录 |
| 进阶训练者 | 使用增力/增肌周期，关注 1RM、RPE/RIR 和减量 |
| 私教/小团队 | 后续分配计划、查看完成情况 |
| 新手用户 | 套用模板，降低训练计划理解成本 |

## 3. 核心业务流程

- 首次使用：seed 系统方案和默认用户计划 -> 创建默认小组 -> 添加成员 -> 输入 1RM -> 进入今日训练。
- 今日训练：读取小组当前用户计划/周期/周数 -> 按星期生成动作 -> 按恢复状态过滤 -> 按成员计算建议重量。
- 训练执行：创建 session -> 生成动作记录和 set -> 多人按动作轮换记录 -> 即时保存 SQLite。
- 动作替换：选择替代动作 -> 保存替代后的 exercise_id，同时保留 replaced_from_exercise_id。
- 训练总结：完成 session -> 统计完成度 -> 生成 progression_suggestions -> 进入总结页。
- 进阶建议：增力看完成情况和 RPE，增肌看次数区间、RIR 和动作质量。

## 5. 页面结构

- 底部 Tab：首页、计划、记录、我的。
- 首页：今日训练任务中心，展示今天练什么、今日重点、当前搭子、本周简要概览和开始训练入口。
- 计划页：当前计划仪表盘、阶段进度、本周安排、我的计划管理、系统方案、创建计划、导入计划和导出计划。
- 记录页：训练历史、PR、预估 1RM、趋势、疲劳提示、最近训练和分析建议。
- 我的页：深色账号卡 + 训练档案、小组成员、偏好设置、账号设置四个主入口 + 关于练刻 + 退出登录；HeroCard 小组和计划区域可点击跳转。关于练刻页面包含关于练刻、意见反馈、用户协议、隐私政策。
- 训练执行页：从首页进入，不显示底部 Tab，使用沉浸式深色全屏页面，突出当前动作、多成员记录卡、重量/次数 stepper、RPE/RIR 快捷输入、完成本组和自动保存状态。
- 总结页：完成度、本次训练总量、成员表现、预估 1RM、本次最佳动作、训练时长、下次建议和恢复建议。

## 6. 权限规则

第一版为本地单设备应用，不做账号登录和云端权限。数据模型保留角色：

- owner：组长，管理小组和计划。
- member：参与训练和记录。
- coach：后续教练模式预留。
- guest：后续只读查看预留。

## 7. 状态机

### 训练 session 状态

```text
draft -> in_progress -> completed
draft -> cancelled
in_progress -> cancelled
```

### 计划周期状态

```text
strength -> deload -> hypertrophy -> deload -> strength
```

### 进阶建议类型

```text
increase | maintain | decrease | deload | add_reps | maintain_or_decrease
```

### 恢复建议

```text
normal | remove_c | only_a | reduce_weight | deload | rest
```

## 8. 边界条件

- 用户跳过 1RM 时仍可进入 App，建议重量显示未设置。
- 周五默认休息，可选补弱；不练周五不需要补回来。
- 状态一般删除 C 动作，状态差只保留 A 动作，状态很差建议休息或减量。
- 器械被占时替换动作，但原计划动作不变。
- 历史记录必须保存当时的计划快照。
- 历史推算只能作为训练建议，不能写成医疗、伤病或绝对结论。
- `.liftmark.json` / `.liftmark` 计划文件不能包含训练记录、成员 1RM 或个人参数。
- 系统方案不是用户计划；用户必须点击“使用此方案”后才生成自己的训练计划。
- 训练记录不能直接绑定系统方案，只能绑定用户计划或训练时计划快照。

## 9. 异常流程

- seed 失败：提示重试，不进入半初始化状态。
- SQLite 写入失败：训练执行页显示错误并保留当前输入状态。
- 缺少成员：今日训练提示添加成员。
- 缺少计划：提示从系统方案复制、创建计划或导入计划。
- 无替代动作：显示空状态并允许返回。
- 导入计划版本不兼容：显示中文提示，不覆盖本地计划。
- 危险设置操作：必须二次确认。

## 10. 业务限制

- 计划是数据，不是代码。
- 计划模板和个人参数分离。
- 计划和训练记录分离。
- 多人逻辑从第一版就保留。
- 本地 SQLite 优先。
- Domain 层不依赖 UI。
- 训练记录不能用 AsyncStorage 保存。
- Excel 训练计划只整理为后续 seed 数据设计说明，不硬编码进页面组件。
