# Workout 模块原理文档

更新时间：2026-06-08

## 1. 核心设计思想

创建训练 session、生成动作记录和每位成员的 set 记录，支撑训练现场多人轮换记录。

模块设计应遵守项目核心原则：

- 计划是数据，不是代码。
- 计划模板和个人参数分离。
- 计划和训练记录分离。
- 多人逻辑从第一版就保留。
- 云端优先，本地 SQLite 作为缓存与离线副本。
- Domain 层不依赖 UI。
- 训练记录不能用 AsyncStorage 保存。
- Excel 训练计划只整理为后续 seed 数据设计说明，不硬编码进页面组件。

## 2. 关键机制

- WorkoutSession、WorkoutExerciseRecord、WorkoutSet。
- 从今日训练生成 session、exercise records、workout sets。
- 训练执行页默认按动作轮换，当前动作下按成员记录 set。
- 每次修改立即保存 SQLite，离开页面后可恢复。

## 3. 设计原因

- 多人训练要求从第一版就保留 `group_id`、`member_id`、`session_id`、`plan_id`。
- Excel 训练计划验证了业务逻辑，但 App 需要把计划拆成可配置数据和可测试算法。
- 本地训练现场要求离线可用、快速输入、即时保存。

## 4. 历史原因

项目源于多人共用 Excel 训练计划，但 Excel 在训练现场切 Sheet、输入和多人记录都不顺手。因此 App 不应复刻表格，而应抽象为计划模板、个人参数、训练记录和进阶建议。

## 5. 约束条件

- 训练记录写 SQLite。
- UI 不写核心业务逻辑。
- 计划模板不与个人参数混合。
- 历史记录保存计划快照。

## 6. 安全考虑

- 本地训练数据包含个人身体和训练表现，导出和清空都需要明确用户确认。
- 后续云同步必须处理权限、冲突和删除。

## 7. 性能考虑

- 训练执行期间操作必须快速响应。
- 派生计算应优先纯函数。
- 历史列表需避免一次性渲染过多记录。

## 8. 不能随便修改的地方

- 训练现场输入必须低摩擦，不能频繁要求键盘输入。
- 训练进行中退出 App 不能丢 set 记录。
- 计划修改不能回写破坏已完成历史。

## 9. 常见误区

- 把 Excel Sheet 当成 UI 页面结构。
- 把计划动作写在组件里。
- 只支持一个成员后再补多人。
- 用 AsyncStorage 保存训练记录。
