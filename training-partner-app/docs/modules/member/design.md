# Member 模块设计文档

更新时间：2026-06-15

## 1. 设计目标

让 `member` 模块拥有清晰边界，支撑多人训练 MVP，并为后续云同步、计划编辑器或商业版扩展保留空间。

## 2. 模块职责

- 成员基础信息：昵称、角色、头像预留。
- 成员训练参数：体重、卧推/深蹲/硬拉/肩推 1RM、引体参考总负重。
- 成员加重单位：杠铃默认 2.5kg，哑铃默认 2kg。
- 成员资料的本地持久化和编辑表单。

## 3. 非职责

- 不决定小组当前计划和当前周数。
- 不生成今日训练内容。
- 不直接计算进阶建议，只提供个人参数。

## 4. 核心业务规则

- 1RM 为空时不得阻塞进入 App，但建议重量必须显示“未设置 1RM”。
- 第一版一个本地默认小组支持最多 5 个成员；少于 2 个成员时仍允许先保存，今日训练再提示补齐。
- 达到 5 人后，新增成员入口应禁用或变为说明卡，文案说明“本地小组最多支持 5 位训练成员，适合一台设备多人轮换记录”。
- 成员数量上限必须从 `src/config/appLimits.ts` 读取，避免页面和校验层写出不同数字。
- 新增成员保存成功后自动返回成员列表；从设置页成员入口进入时返回 `settings/members`，从搭子页进入时返回搭子页。
- 成员表单保存体重、卧推/深蹲/硬拉/肩推 1RM、引体参考重量、杠铃加重单位和哑铃加重单位。
- 成员删除策略待确认，不能破坏历史训练记录。

## 5. 状态流转

- 本模块状态必须通过 Repository 写入 SQLite 或由 Domain 纯函数派生。
- 涉及训练执行的数据不得只存在组件局部状态。

## 6. 权限规则

第一版为本地默认小组，不做账号鉴权。模型和 Repository 需保留后续 owner/member/coach/guest 扩展空间。

## 7. 输入输出

- 输入：来自 UI 表单、SQLite Repository、seed 数据或其他 Domain 模块。
- 输出：领域对象、派生结果、Repository 持久化结果或 UI 可展示 ViewModel。

## 8. 异常处理

- 输入缺失时返回可解释状态，不让 UI 崩溃。
- SQLite 写入失败必须上抛或返回错误结果。
- 不允许静默丢弃训练现场输入。

## 9. 边界条件

- Sprint 2 已创建成员列表、新增和编辑基础实现；本文档仍约束后续业务实现。
- 后续代码落地后，需用源码验证本文档并修正不一致处。

## 10. 扩展点

- Repository 接口可增加远程同步实现。
- Domain 函数应保持可测试、可复用。
- seed 数据可替换为用户自定义计划或导入计划。

## 11. 修改风险

- 1RM 为空时不得阻塞进入 App，但建议重量必须显示“未设置 1RM”。
- 成员删除策略待确认，不能破坏历史训练记录。
