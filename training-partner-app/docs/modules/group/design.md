# Group 模块设计文档

## 2026-06-30 小组动作分析规则补充

- 小组动作表现卡可以进入动作详情页；详情页按成员拆分展示最好重量、容量、预估 1RM 和最近有效组。
- 动作详情页必须提供指标、时间范围和成员筛选，不用单一固定榜单替代用户分析。
- 多成员趋势线只使用本机训练记录推导，不填充假数据。
- 小组记录默认展示小组总览，动作表现必须通过选择器进入。
- 动作选择器优先列出真实练过的动作，不使用旧的固定默认动作作为入口。

## 2026-06-30 多小组规则补充

- 小组页提供当前小组切换器和创建新小组入口。
- 创建新小组复用当前小组的当前计划、阶段、周数和周五设置，但不自动复制成员。
- 当前小组影响今日训练、成员页、记录页、设置页、头像同步和身体数据页。
- 切换小组只改变新操作和视图上下文，不迁移历史 session 的 `group_id`。

## 2026-06-29 本地小组记录规则补充

- 当前版本的小组是本地小组，适合同一台设备多人轮换记录。
- 数据保存在本机 SQLite。
- 当前版本组长可以查看本机保存的所有本地成员训练数据。
- 记录页支持个人 / 小组切换：个人统计当前成员数据；小组统计本机小组训练量、成员贡献、完成率、近期训练和近 7 天折线趋势。
- 小组统计必须由 `workout_sessions`、`workout_exercise_records` 和 `workout_sets` 推导，不能使用展示假数据。
- 多设备小组能力后续版本开放；当前主界面不展示成员邀请或二维码加入入口。
- 搭子页主页面只保留一句短说明，完整规则放在“了解本地小组”二级弹层中，避免主页面被长说明占据。

## 1. 设计目标

让 `group` 模块拥有清晰边界，支撑多人训练 MVP，并为后续云同步、计划编辑器或商业版扩展保留空间。

## 2. 模块职责

- 默认小组创建和读取。
- 多小组列出、创建和当前小组切换。
- 小组 activePlanId、currentPhaseType、currentWeek、fridayEnabled。
- 小组内多人训练的边界和未来 owner/coach/member/guest 角色预留。

## 3. 非职责

- 不保存成员 1RM 细节。
- 不保存训练每组记录。
- 不实现成员邀请和二维码加入，第一版只预留字段和 Repository 边界。

## 4. 核心业务规则

- 第一版即使只有一个默认小组，也不能把 UI 和数据结构做成单人逻辑。
- currentWeek 和 currentPhaseType 必须与 plan phase 范围保持一致。

## 5. 状态流转

- 本模块状态必须通过 Repository 写入 SQLite 或由 Domain 纯函数派生。
- 涉及训练执行的数据不得只存在组件局部状态。

## 6. 权限规则

第一版为本地默认小组，不做账号鉴权。模型和 Repository 需保留后续 owner/member/coach/guest 扩展空间。

## 7. 输入输出

- 输入：来自 UI 表单、SQLite Repository、seed 数据或其他 Domain 模块。
- 输出：领域对象、派生结果、Repository 持久化结果或 UI 可展示 ViewModel。

## 8. 异常处理

- 输入缺失时返回可解释状态，不让 UI 崩溃。
- SQLite 写入失败必须上抛或返回错误结果。
- 不允许静默丢弃训练现场输入。

## 9. 边界条件

- Sprint 1 已创建基础源码骨架；本文档仍约束后续业务实现。
- 后续代码落地后，需用源码验证本文档并修正不一致处。

## 10. 扩展点

- Repository 接口可增加远程同步实现。
- Domain 函数应保持可测试、可复用。
- seed 数据可替换为用户自定义计划或导入计划。

## 11. 修改风险

- 第一版即使只有一个默认小组，也不能把 UI 和数据结构做成单人逻辑。
- currentWeek 和 currentPhaseType 必须与 plan phase 范围保持一致。
