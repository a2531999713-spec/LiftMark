# History 模块设计文档

更新时间：2026-06-14

## 2026-06-14 最近训练摘要卡

- 最近训练列表改为“训练摘要卡”，不再呈现表单感或后台列表感。
- 每条卡片展示训练标题/训练日、日期和时长、完成动作数、完成组数、总训练量、当前成员口径、PR/估算 1RM/趋势标签，并通过右侧箭头进入详情。
- 空状态文案为“还没有训练记录 / 完成一次训练后，这里会显示训练摘要、训练量和进步趋势”，提供“去训练”按钮。
- 点击训练卡进入 `app/history/[sessionId].tsx`；历史详情仍展示训练当时快照，不受后续计划切换影响。

## 1. 设计目标

让 `history` 模块拥有清晰边界，支撑多人训练 MVP，并为后续云同步、计划编辑器或商业版扩展保留空间。

## 2. 模块职责

- 训练历史列表和训练详情读取。
- 按成员/动作筛选。
- 每个动作最近一次记录、历史最好表现、预估 1RM 展示。
- 下次建议展示。
- 最近 5 次趋势、PR 接近度和疲劳提示。

## 3. 非职责

- 不生成原始训练记录。
- 不修改计划模板。
- 不作为数据分析大盘，复杂图表第一版延后。

## 4. 核心业务规则

- 历史记录必须展示训练当时的计划快照，而不是当前计划。
- 第一版复杂图表延后，避免拖慢 MVP。
- 历史推算是训练建议，不是医疗或伤病判断。

## 5. 状态流转

- 本模块状态必须通过 Repository 写入 SQLite 或由 Domain 纯函数派生。
- 涉及训练执行的数据不得只存在组件局部状态。

## 6. 权限规则

第一版为本地默认小组，不做账号鉴权。模型和 Repository 需保留后续 owner/member/coach/guest 扩展空间。

## 7. 输入输出

- 输入：来自 UI 表单、SQLite Repository、seed 数据或其他 Domain 模块。
- 输出：领域对象、派生结果、Repository 持久化结果或 UI 可展示 ViewModel。

## 8. 异常处理

- 输入缺失时返回可解释状态，不让 UI 崩溃。
- SQLite 写入失败必须上抛或返回错误结果。
- 不允许静默丢弃训练现场输入。

## 9. 边界条件

- Sprint 1 已创建基础源码骨架；本文档仍约束后续业务实现。
- 后续代码落地后，需用源码验证本文档并修正不一致处。

## 10. 扩展点

- Repository 接口可增加远程同步实现。
- Domain 函数应保持可测试、可复用。
- seed 数据可替换为用户自定义计划或导入计划。

## 11. 修改风险

- 历史记录必须展示训练当时的计划快照，而不是当前计划。
- 第一版复杂图表延后，避免拖慢 MVP。
