# Recovery 模块原理文档

更新时间：2026-06-30

## 1. 核心设计思想

根据睡眠、食欲、训练欲望、酸痛、关节不适和疲劳生成恢复建议；今日训练页把同一 A/B/C 过滤规则呈现为“动作筛选”。

模块设计应遵守项目核心原则：

- 计划是数据，不是代码。
- 计划模板和个人参数分离。
- 计划和训练记录分离。
- 多人逻辑从第一版就保留。
- 云端优先，本地 SQLite 作为缓存与离线副本。
- Domain 层不依赖 UI。
- 训练记录不能用 AsyncStorage 保存。
- Excel 训练计划只整理为后续 seed 数据设计说明，不硬编码进页面组件。

## 2. 关键机制

- RecoveryLog 数据模型。
- 恢复评分 totalScore 和 recommendation。
- normal、remove_c、only_a、reduce_weight、deload、rest 建议。
- 动作筛选对动作优先级过滤的规则。

## 3. 设计原因

- 多人训练要求从第一版就保留 `group_id`、`member_id`、`session_id`、`plan_id`。
- Excel 训练计划验证了业务逻辑，但 App 需要把计划拆成可配置数据和可测试算法。
- 本地训练现场要求离线可用、快速输入、即时保存。

## 4. 历史原因

项目源于多人共用 Excel 训练计划，但 Excel 在训练现场切 Sheet、输入和多人记录都不顺手。因此 App 不应复刻表格，而应抽象为计划模板、个人参数、训练记录和进阶建议。

## 5. 约束条件

- 训练记录写 SQLite。
- UI 不写核心业务逻辑。
- 计划模板不与个人参数混合。
- 历史记录保存计划快照。

## 6. 安全考虑

- 本地训练数据包含个人身体和训练表现，导出和清空都需要明确用户确认。
- 后续云同步必须处理权限、冲突和删除。

## 7. 性能考虑

- 训练执行期间操作必须快速响应。
- 派生计算应优先纯函数。
- 历史列表需避免一次性渲染过多记录。

## 8. 不能随便修改的地方

- 关节不适高分应优先休息或只做 A，不能硬推训练。
- 恢复评分输入可能缺失，今日训练应有默认模式。

## 9. 常见误区

- 把 Excel Sheet 当成 UI 页面结构。
- 把计划动作写在组件里。
- 只支持一个成员后再补多人。
- 用 AsyncStorage 保存训练记录。
