# 练刻 LiftMark 云服务与在线多人同练完整设计方案

版本：v1.0

方案类型：账号系统、用户可控云同步、全量数据恢复、在线多人同练、低成本服务器、PC 灾备、后台运营

适用产品：练刻 LiftMark

核心目标：用户登录账号后，可以选择是否上传数据；开启云同步后，卸载、重装或换手机登录后数据不会丢失；在线多人同练中，队友状态在正常网络下 1-3 秒内更新。
47.100.239.29(公)
172.24.34.29(私有)

## 1. 方案总览

练刻采用以下完整云服务架构：

```text
本地优先训练
+ 用户可控云同步
+ 全量结构化数据恢复
+ 低成本单服务器
+ 开发者 PC 长期备份
+ 在线多人同练 1-3 秒准实时
```

核心原则：

```text
手机本地 SQLite 是训练现场主数据库。
服务器数据库是账号数据和同步数据的云端副本。
用户可以选择是否上传训练数据。
开启云同步后，卸载、重装或换手机登录后可以恢复完整训练环境。
关闭云同步后，训练数据只保存在当前设备，卸载或换手机后未上传数据无法自动恢复。
在线多人同练不要求毫秒级实时，只要求正常网络下 1-3 秒内看到队友状态变化。
开发者 PC 每天备份服务器数据，作为低成本灾备方案。
```

最终用户体验目标：

```text
用户在旧手机使用练刻训练。
用户登录账号并开启云同步。
成员、计划、小组、训练记录、设置和会员权益同步到服务器。
用户卸载重装或换新手机。
重新安装练刻并登录账号。
App 自动发现云端数据。
用户确认恢复后，手机恢复到接近原来的使用状态。
```

## 4. 本地优先原则

练刻训练现场必须保持本地优先。

所有训练关键操作都必须先写入手机本地 SQLite：

```text
完成一组
修改重量
修改次数
选择 RPE / RIR
跳过动作
替换动作
结束训练
补录训练
编辑历史记录
```

正确流程：

```text
用户完成一组
  ↓
立即保存本地 SQLite
  ↓
写入同步队列
  ↓
UI 显示已保存
  ↓
后台上传服务器
  ↓
服务器保存结构化数据
  ↓
如为在线同练，广播给房间内其他成员
```

禁止流程：

```text
用户完成一组
  ↓
请求服务器
  ↓
服务器成功
  ↓
才算保存成功
```

原因：

```text
健身房网络可能很差。
训练中用户可能切屏、锁屏或 App 被系统杀掉。
训练记录不能因为服务器异常而丢失。
训练执行页必须低延迟。
```

## 5. 用户上传数据规则

用户登录后，系统主动提示是否开启云同步。

### 5.1 登录后提示

首次登录后弹出提示：

```text
开启云同步？
开启后，练刻会将你的成员档案、训练计划、训练记录和训练进度同步到云端。
你卸载重装、换手机或重新登录后，可以恢复数据。
你可以随时在设置中关闭云同步。
```

按钮：

```text
开启云同步
暂不开启
```

### 5.2 用户选择结果

| 用户选择 | 数据行为 | 恢复能力 |

|---|---|---|

| 开启云同步 | 本机数据和后续数据同步到服务器 | 卸载、重装、换手机后可恢复 |

| 暂不开启 | 数据只保存在当前手机 | 卸载、清空数据、换手机后无法自动恢复 |

| 后续开启 | 上传当前本机已有数据，并继续同步新增数据 | 开启后可恢复已上传数据 |

| 后续关闭 | 新增数据只保存在本机 | 关闭后未上传数据无法自动恢复 |

### 5.3 关闭云同步提示

用户关闭云同步时必须提示：

```text
关闭云同步后，新产生的训练数据只会保存在当前设备。
卸载 App、清空数据或更换手机后，未上传的数据将无法自动恢复。
```

按钮：

```text
确认关闭
继续开启
```

### 5.4 云同步不是强制上传

用户可以不上传训练数据。
但产品必须明确：

```text
想实现卸载重装、换手机后数据不丢，需要开启云同步。
如果用户选择不上传数据，练刻无法保证卸载或换机后恢复本机训练记录。
```

## 7. 账号与成员关系

练刻必须区分账号和训练成员。

```text
账号 user：登录身份，用于会员、同步、权限、恢复。
成员 member：训练个体，用于 1RM、训练记录、计划执行。
```

原因：

```text
一台手机可以记录多人训练。
本机成员可能没有账号。
账号用户可以拥有多个训练成员档案。
本机成员后续可以被真实账号认领。
```

### 7.1 成员类型

| 类型 | 说明 |

|---|---|

| 本机成员 | 没有登录账号，只存在当前设备或创建者账号下 |

| 账号成员 | 已绑定用户账号，可跨设备同步 |

| 临时成员 | 只参与某次训练，默认不长期同步到他人账号 |

| 被认领成员 | 原本是本机成员，后续由真实账号确认绑定 |

### 7.2 本机成员认领

场景：

```text
A 的手机记录了 B 的训练数据。
B 后续注册账号。
B 希望把历史数据认领到自己的账号。
```

流程：

```text
A 打开成员管理
选择 B 的本机成员档案
发起认领邀请
B 登录账号并确认这是自己的历史记录
系统将该 member_profile 绑定到 B 的 user_id
后续 B 可以在自己的手机恢复该成员数据
```

规则：

```text
本机成员不会自动归属其他账号。
认领必须由对方账号确认。
认领后历史记录归属对应成员。
小组汇总记录仍保留小组关系。
```

## 9. 同步状态设计

每条核心数据都应有同步状态。

### 9.1 状态定义

| 状态 | 说明 |

|---|---|

| local_only | 仅本地保存，未开启云同步或未登录 |

| pending_sync | 等待上传 |

| syncing | 正在同步 |

| synced | 已同步 |

| sync_failed | 同步失败，等待重试 |

| conflict | 存在冲突，需要处理 |

| deleted_pending | 本地已删除，等待云端同步删除 |

### 9.2 页面显示文案

| 状态 | 用户文案 |

|---|---|

| local_only | 当前仅本机保存 |

| pending_sync | 待同步 |

| syncing | 同步中 |

| synced | 已同步 |

| sync_failed | 同步失败，稍后自动重试 |

| conflict | 有数据冲突，请处理 |

| deleted_pending | 删除同步中 |

### 9.3 设置页同步入口

设置页需要显示：

## 10. 同步触发时机

同步不要高频到影响训练，也不能太低频导致数据长时间不上传。

触发条件：

```text
用户登录成功
App 启动
App 回到前台
网络从断开变为可用
完成一次训练后
修改成员后
修改计划后
结束在线同练后
用户手动点击立即同步
每天自动同步一次
```

训练中策略：

```text
完成一组后本地保存。
同步事件进入队列。
训练中可每 30-60 秒批量同步一次。
训练结束后立即同步一次。
网络差时全部进入 pending_sync。
```

在线同练中策略：

```text
最终训练数据走同步队列。
房间状态走准实时事件通道。
两者互不替代。
```

## 11. 冲突处理设计

同步冲突不可避免，尤其是多设备登录、离线训练、计划编辑、小组协作场景。

### 11.1 训练记录冲突

原则：

```text
不轻易覆盖训练记录。
训练记录以 session_id、exercise_record_id、set_id 合并。
重复上传要去重。
同一 set 多次编辑，以最新 updated_at 为准。
冲突无法判断时保留两份，提示用户处理。
```

### 11.2 成员参数冲突

例如体重、1RM、加重单位。

规则：

```text
默认以后修改的一方为准。
保留更新时间。
必要时保留历史变更记录。
```

### 11.3 计划冲突

计划结构冲突不能直接覆盖。

规则：

```text
如果同一计划在两个设备都被修改，生成冲突副本。
用户可以选择保留本机版本、云端版本或另存为新计划。
```

### 11.4 删除冲突

删除采用软删除。

规则：

```text
本地删除 -> 标记 deleted_at -> 同步云端。
云端删除 -> 下发 deleted_at -> 本地隐藏。
不立即物理删除。
保留一定恢复窗口。
```

## 15. 在线多人同练完整设计

在线多人同练是完整云服务的一部分，不是额外扩展。

目标：

```text
小组成员各自用自己的手机进入同一场训练。
每个人记录自己的数据。
其他成员在正常网络下 1-3 秒内看到状态变化。
断网时本地继续训练。
恢复网络后自动补同步。
```

## 21. 1-3 秒状态同步机制

采用准实时同步：

```text
优先实时连接
实时连接失败后 3 秒轮询兜底
每 10-15 秒拉取房间快照校正
```

正常情况：

```text
用户完成一组
  ↓
服务器收到事件
  ↓
服务器推送给房间成员
  ↓
其他成员约 1 秒内看到变化
```

实时连接失败：

```text
客户端每 3 秒拉取新事件
  ↓
根据 last_event_seq 补齐缺失事件
```

长时间异常：

```text
客户端每 10-15 秒拉取房间快照
  ↓
校正当前动作、成员状态和完成进度
```

目标：

```text
正常网络：1-3 秒内看到队友变化。
弱网：允许延迟。
断网：本地继续训练。
恢复网络：自动补同步。
```

## 32. 数据量估算

前期只同步结构化训练数据，不上传视频，服务器空间压力可控。

## 35. 最终产品规则文案

可以直接写入产品文档：

```text
练刻支持账号云同步。
用户登录后可以选择是否开启云同步。开启后，练刻会将成员档案、小组、训练计划、动作、训练记录、训练进度、训练设置和会员权益同步到云端。
训练时数据仍然先保存到手机本地，离线也能继续训练。网络恢复后，App 会自动同步待上传数据。
用户卸载重装、换手机或重新登录后，只要开启过云同步，就可以恢复云端数据，尽量保持和原设备一致。
如果用户关闭云同步，训练数据只保存在当前设备。卸载 App、清空数据或更换手机后，未上传的数据无法自动恢复。
练刻在线多人同练采用训练房间机制。小组成员进入同一房间后，各自记录自己的训练数据。正常网络下，队友状态会在 1-3 秒内更新。网络异常时，用户仍可继续训练，数据保存在本机；网络恢复后自动补同步。
开发者 PC 会定期备份服务器数据库，用于服务器故障时的数据恢复。
```

## 36. 最终结论

练刻云服务最终采用：

```text
本地优先
+ 用户可控云同步
+ 全量结构化数据恢复
+ 低成本单服务器
+ PC 灾备
+ 在线多人同练准实时房间
```

一句话：

```text
用户训练时先写手机 SQLite，保证离线稳定；用户开启云同步后，所有核心训练数据同步到服务器；卸载、重装或换手机登录后可恢复；在线多人同练通过训练房间和事件同步实现 1-3 秒队友状态更新；服务器用低成本单机起步，开发者 PC 每天备份数据库作为灾备。
```
